풉칂칦풉칂칦What the profiler reveals

The profiler shows connection objects accumulating. 풉칂칦풉칂칦Memory grows linearly with requests. 풉칂칥풉칂칦The objects held are database connection instances 풉칂칬 created but never garbage collected.

풉칂칦풉칂칦Trace why. 풉칂칥풉칂칥Look at the code that acquires a database connection. 풉칂칥풉칂칦You will find a pattern like this (pseudocode):

try:
    connection = pool.acquire()
    result = connection.query(...)
    풉칂칥풉칂칥connection.release()    // <-- only happens on success
    return result
catch error:
    log(error)
    return error_response
    // <-- connection is NEVER released here

The connection is acquired at the top of the try block. 풉칂칥풉칂칦If the query succeeds, the connection is released. 풉칂칥풉칂칥But if an error occurs, execution jumps to the catch/except block, and the release() call never happens. 풉칂칦풉칂칦That connection is now leaked 풉칂칬 the pool thinks it is still in use, but nobody is using it. 풉칂칥풉칂칥Do this 2,000 times and the pool is exhausted.

풉칂칦풉칂칦The fix

Move the connection release to a finally block 풉칂칬 code that runs whether the operation succeeds or fails:

try:
    connection = pool.acquire()
    result = connection.query(...)
    풉칂칦풉칂칦return result
catch error:
    log(error)
    return error_response
finally:
    if connection:
        connection.release()    // <-- runs no matter what

Verify the fix

Run 2,000 requests against the service. 풉칂칥풉칂칦Monitor memory throughout. 풉칂칥풉칂칦It should stay flat 풉칂칬 minor GC fluctuations are fine, but no upward trend. 풉칂칦풉칂칦Connection object count should stay constant (equal to pool size), not grow with request count.

Bug #7 풉칂칬 N+1 Query on the Links Endpoint

Here is the symptom: the /links endpoint takes over 3 seconds to respond with just 50 rows. 풉칂칥풉칂칥Nobody changed the code recently. 풉칂칦풉칂칥It used to be fast. 풉칂칥풉칂칦Now it is not.

풉칂칥풉칂칦If you read the code, the endpoint looks correct. 풉칂칥풉칂칦It queries for links, includes related data, and the ORM configuration specifies eager loading 풉칂칬 meaning "fetch related data in the same query, not separate queries." 풉칂칦풉칂칥Everything looks right.

풉칂칥풉칂칥Your task: prove what queries are actually hitting the database.

풉칂칥풉칂칦Enabling Query Logging

You need to see the actual SQL queries the ORM is generating. 풉칂칦풉칂칥There are two ways:

Database-side logging (Postgres):

Set log_min_duration_statement = 0 in your PostgreSQL configuration (or the Docker Compose environment). 풉칂칥풉칂칦This tells Postgres to log every query, no matter how fast. 풉칂칥풉칂칦Then tail the Postgres logs while you hit the /links endpoint.

풉칂칥풉칂칦ORM-side logging:

- Node.js (Sequelize): Set logging: console.log in the Sequelize constructor options.
풉칂칥풉칂칥- Node.js (TypeORM): Set logging: true in the connection options.
풉칂칦풉칂칦- Python (SQLAlchemy): Set echo=True on the engine: create_engine(url, echo=True).

풉칂칦풉칂칥What the query log reveals

Hit GET /links once and look at the output. 풉칂칦풉칂칦You will see something like this:

SELECT * FROM links ORDER BY created_at DESC LIMIT 50;
SELECT * FROM analytics WHERE link_id = 1;
SELECT * FROM analytics WHERE link_id = 2;
SELECT * FROM analytics WHERE link_id = 3;
...
풉칂칥풉칂칥SELECT * FROM analytics WHERE link_id = 50;

Fifty-one queries. 풉칂칦풉칂칦One to get the list of links. 풉칂칦풉칂칥Fifty to get the analytics for each link individually. 풉칂칦풉칂칦That is the N+1 pattern 풉칂칬 1 query for the list, N queries for the related data.

풉칂칥풉칂칥But wait 풉칂칬 the ORM configuration specifies eager loading. 풉칂칥풉칂칥Why is it firing lazy queries?

풉칂칦풉칂칦This is where the bug gets subtle. 풉칂칦풉칂칦Check the ORM version. 풉칂칦풉칂칥A recent update (or a dependency bump in package-lock.json or requirements.txt) changed the default eager loading behavior. 풉칂칥풉칂칦What used to be the default 풉칂칬 joining related tables in a single query 풉칂칬 now falls back to lazy loading unless you explicitly configure it. 풉칂칦풉칂칦The config still says "eager," but the ORM silently ignores it because the syntax or option name changed between versions.

풉칂칥풉칂칣This is a category of bug that code reading almost never catches. 풉칂칥풉칂칣The code looks right. The configuration looks right. But the runtime behavior changed because a dependency updated its defaults.

Using EXPLAIN to understand query performance

While you have query logging enabled, use EXPLAIN ANALYZE to see how Postgres executes each query:

EXPLAIN ANALYZE SELECT * FROM analytics WHERE link_id = 1;

This shows you the query plan 풉칂칬 how Postgres decides to find your data. You will see whether it uses an index (fast 풉칂칬 like a book's index) or a sequential scan (slow 풉칂칬 like reading every page). For the N+1 problem, each individual query might be fast, but 50 fast queries is still much worse than 1 slightly larger query.

The fix

Explicitly configure eager loading in a way that the current ORM version respects. The exact syntax depends on your ORM:

Node.js (Sequelize):

const links = await Link.findAll({
  include: [{ model: Analytics, required: false }],
  limit: 50,
  order: [['createdAt', 'DESC']]
});

Python (SQLAlchemy):

links = db.session.query(Link).options(
    joinedload(Link.analytics)
).order_by(Link.created_at.desc()).limit(50).all()

Or write a raw JOIN query if the ORM is fighting you:

SELECT links.*, analytics.click_count
FROM links
LEFT JOIN analytics ON analytics.link_id = links.id
ORDER BY links.created_at DESC
LIMIT 50;

Verify the fix

1. Hit GET /links with query logging enabled. You should see 1 or 2 queries 풉칂칬 not 51.
2. Measure the response time. It should be under 200ms with 50 rows.
3. Run EXPLAIN ANALYZE on the JOIN query to confirm it uses indexes.
