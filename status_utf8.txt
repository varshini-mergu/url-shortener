STEP 3: BUILD -- The Iteration Challenge

Code generation starts now.

풉칂칣풉칂칦풉칂칣Now you apply your strategy to a real task. 풉칂칦풉칂칦풉칂칦You are building the real-time activity feed with WebSocket updates -- the piece of the Team Collaboration feature that pushes live events to connected clients.

풉칂칦풉칂칦풉칂칦This task is deliberately chosen. 풉칂칦풉칂칦풉칂칦Real-time features are where AI agents struggle the most. 풉칂칦풉칂칦풉칂칥They require correct handling of connection lifecycle (open, close, error, reconnect), concurrent state (multiple clients, multiple channels), and event ordering. 풉칂칦풉칂칥풉칂칦AI agents tend to produce demos that work for one client in a happy-path scenario and fall apart under real conditions.

풉칂칥풉칂칦풉칂칥Expect your first output to need iteration. 풉칂칥풉칂칥풉칂칦That is the point.

풉칂칥풉칂칦풉칂칦The Iteration Log

Before you write your first prompt, set up your iteration log. 풉칂칦풉칂칥풉칂칥You will track every iteration with this format:

Iteration: [1, 2, 3, or 4]
Type: [initial | refinement | restart]
Structural quality: [1-5]
Surface quality: [1-5]
Key issues found: [list]
Decision: [refine | restart | ship]
Reasoning: [why this decision]

Structural quality rubric:

- 1: Wrong approach entirely. 풉칂칥풉칂칥풉칂칦Cannot be salvaged.
풉칂칦풉칂칦풉칂칦- 2: Partially right idea, but critical architectural flaw (wrong pattern, wrong boundaries).
풉칂칥풉칂칦풉칂칦- 3: Sound approach with one significant structural gap (missing reconnection model, no channel isolation).
풉칂칦풉칂칦풉칂칦- 4: Good architecture with minor structural concerns (could handle edge cases better, scaling story is unclear).
풉칂칥풉칂칦풉칂칥- 5: Clean architecture, correct patterns, would pass an architecture review.

풉칂칦풉칂칥풉칂칦Surface quality rubric:

- 1: Barely functional. 풉칂칦풉칂칦풉칂칥Major gaps in implementation.
풉칂칦풉칂칥풉칂칦- 2: Partially implemented. 풉칂칥풉칂칦풉칂칥Multiple missing pieces (stubs, TODOs, incomplete error handling).
풉칂칥풉칂칥풉칂칦- 3: Functional but rough. 풉칂칥풉칂칦풉칂칦Works for happy path, missing edge cases and polish.
풉칂칦풉칂칦풉칂칦- 4: Good implementation. 풉칂칥풉칂칦풉칂칥Handles most cases, minor naming or convention issues.
풉칂칦풉칂칥풉칂칦- 5: Production-ready. 풉칂칥풉칂칥풉칂칥Clean code, proper error handling, follows all conventions.

풉칂칦풉칂칥풉칂칦Round 1: The Initial Prompt

Write your prompt for the real-time activity feed. 풉칂칦풉칂칥풉칂칥Include:

- What the feature does (push team activity events to connected clients via WebSocket)
- Architectural context from your codebase (existing patterns, conventions, middleware)
- Constraints (must handle connection, disconnection, reconnection; must scope events to teams; must integrate with existing auth)
- Expected output structure (which files, what format)

Run the prompt. 풉칂칦풉칂칦풉칂칦Review the output. 풉칂칦풉칂칥풉칂칥Rate it on both dimensions. 풉칂칥풉칂칦풉칂칦Log it.

풉칂칦풉칂칦풉칂칥If your initial output scores structural >= 3: proceed to refinement. 풉칂칦풉칂칦풉칂칦Identify the specific surface issues and write a targeted follow-up prompt for each one (or batch related fixes).

풉칂칥풉칂칥풉칂칥If your initial output scores structural < 3: stop. 풉칂칦풉칂칥풉칂칥Do not refine. 풉칂칥풉칂칦풉칂칥Restart. 풉칂칥풉칂칦풉칂칦Write a new prompt that takes a different approach. 풉칂칥풉칂칥풉칂칦Use what you learned from the first output -- what went wrong structurally -- to inform the new prompt. 풉칂칦풉칂칦풉칂칦Maybe you need to specify the event architecture explicitly. 풉칂칥풉칂칦풉칂칥Maybe you need to include an example of how events flow through the system. 풉칂칦풉칂칦풉칂칦Maybe you need to constrain the agent away from the approach it chose.

풉칂칥풉칂칦풉칂칥Round 2: First Refinement (or Restart)

If refining: write a follow-up prompt that addresses the specific issues from Round 1. 풉칂칦풉칂칥풉칂칥Be surgical. 풉칂칦풉칂칦풉칂칦Do not say "make it better." 풉칂칦풉칂칥풉칂칦Say exactly what is wrong and exactly what the fix should look like. 풉칂칥풉칂칥풉칂칦Example:

Bad follow-up: "The WebSocket handling needs improvement."

풉칂칥풉칂칥풉칂칦Good follow-up: "The WebSocket connection does not handle unexpected disconnections. 풉칂칥풉칂칦풉칂칥Add a heartbeat mechanism that pings every 30 seconds. 풉칂칦풉칂칦풉칂칥If no pong is received within 10 seconds, close the connection and clean up the subscription. 풉칂칥풉칂칥풉칂칥On the client reconnection, resubscribe to the team channel and replay events missed since the last received event timestamp."

풉칂칦풉칂칦풉칂칥Rate the output again. 풉칂칥풉칂칦풉칂칥Log it. 풉칂칦풉칂칥풉칂칦Is quality improving? 풉칂칥풉칂칥풉칂칦Structural score should stay the same or go up. 풉칂칦풉칂칦풉칂칥Surface score should go up. 풉칂칦풉칂칦풉칂칥If either goes down, you have a problem.

풉칂칥풉칂칦풉칂칦If restarting: write a completely new prompt in a fresh conversation. 풉칂칦풉칂칥풉칂칦Incorporate the lessons from Round 1. 풉칂칦풉칂칥풉칂칥Be explicit about the architecture you want. 풉칂칥풉칂칥풉칂칦If the agent chose HTTP polling, specify WebSocket. 풉칂칦풉칂칦풉칂칦If it coupled to the database, specify service-layer events. 풉칂칦풉칂칥풉칂칥If it ignored auth, make auth a hard constraint.

풉칂칦풉칂칦풉칂칦Rate the restart output. 풉칂칥풉칂칣풉칂칥Log it. 풉칂칣This is now your new baseline.

Round 3: Second Refinement

Same process. Targeted follow-up. Rate. Log. Check trajectory.

At this point, apply the two-iteration rule (if that is your strategy). Look at your log. Are scores improving? Flat? Declining? If they are not clearly improving, you have one iteration left and a decision to make.

Round 4: Final Iteration

This is your last shot. Either a final refinement to polish, or a last-resort restart if the trajectory has gone wrong.

After this iteration, you ship whatever you have. Four iterations is the budget. Period.
