풉칂칦풉칂칦Bug #9: Stale Cache Serving Deleted Records

풉칂칦풉칂칦Why would a deleted link still redirect? 풉칂칥풉칂칥Think about every system involved in a redirect: the API receives the request, checks somewhere for the target URL, and returns a 302. 풉칂칥풉칂칥Where is "somewhere"?

풉칂칦풉칂칦Test: Query the database directly. 풉칂칥풉칂칦Run a SELECT against the links table for the deleted link ID. 풉칂칥풉칂칦Is it there or not?

풉칂칥풉칂칦Here is what you will find. 풉칂칦풉칂칦The database does NOT have the link. 풉칂칦풉칂칦Hypothesis 1 is disproven -- the delete worked. 풉칂칥풉칂칦The database did its job. 풉칂칥풉칂칦But Redis DOES have the link, with the full cached redirect URL. 풉칂칥풉칂칦Hypothesis 2 is confirmed.

풉칂칦풉칂칥Bug #10: Queue Retry Storm

The report reads: "The entire API has become unresponsive. 풉칂칦풉칂칦No error messages. 풉칂칥풉칂칥No crashes. 풉칂칥풉칂칥It just... 풉칂칦풉칂칦stopped responding. 풉칂칦풉칂칦Requests hang indefinitely. 풉칂칦풉칂칥The health check times out."

풉칂칥풉칂칦This is the hardest bug you have faced. 풉칂칦풉칂칦Take it seriously. 풉칂칥풉칂칣The symptom -- API unresponsive -- has no obvious connection to the cause. 풉칂칥풉칂칣There are no errors in the API logs because the API is not throwing errors. It is not crashing. It is waiting. Waiting for something it will never get.

Start with your hypotheses. What could make an API hang without errors?

Hypothesis 1: "The API process is stuck or deadlocked." Maybe an infinite loop, a deadlock, or a hanging synchronous operation.

Test: Check the API process. Is it consuming CPU? Is it responsive to signals? Can you get a thread dump or process snapshot? If it were stuck in a loop, CPU would be pegged. If it were deadlocked, a thread dump would show it.

Hypothesis 2: "The API is waiting for a shared resource that is not available." Every API request that touches the database needs a connection from the connection pool. If the pool is exhausted -- every connection is checked out and none are being returned -- then all new requests will queue up, waiting for a connection. No error. No crash. Just... waiting.

Test: Check the connection pool stats. How many connections are in use? How many are available? Is the pool at capacity?

Test Hypothesis 2. Check the connection pool. You will find it is at 100% utilization. Every single connection is in use. Zero available. The API requests are all waiting in line for a database connection that will never come.

But here is the question that separates a competent debugger from an exceptional one: who is holding all the connections?

It is not the API requests -- they are the ones waiting. Something else checked out all the connections and is not returning them. What else in this system uses the database?

The queue worker.

Check the queue worker. What is it doing? It is retrying failed jobs. Look at the retry configuration: no backoff delay (retries happen immediately), no maximum retry count (retries continue forever). A single failed job triggers an infinite loop of immediate retries. Each retry attempt acquires a database connection, hits the same error that caused the original failure, and immediately tries again. It never releases the connection back to the pool before grabbing another one. Within seconds, the queue worker has consumed every available connection.

Here is the chain of causation. Follow it carefully:

1. A queue job fails (for any reason -- bad data, temporary DB hiccup, anything)
2. The queue worker retries immediately (no backoff)
3. Each retry acquires a DB connection from the shared pool
4. The retry fails again (same error)
5. The worker retries again immediately (no max retry count)
6. Steps 3-5 repeat in a tight loop
7. The shared connection pool is exhausted within seconds
8. API requests that need the database cannot get a connection
9. API requests hang indefinitely, waiting for connections
10. The entire API becomes unresponsive
11. Health checks (which also need the database) time out
12. Monitoring reports the API as down

Twelve steps from "one queue job failed" to "the entire API is down." The symptom is at step 12. The root cause is at step 1. The design flaws are at steps 2, 5, and 7 (no backoff, no retry limit, shared connection pool).

The fix has three parts. All three are necessary. Any one alone is insufficient.

1. Add exponential backoff to queue retries. Instead of retrying immediately, wait 1 second, then 2 seconds, then 4, then 8, and so on. This prevents the tight loop that burns through connections. Exponential backoff is a pattern where each retry waits twice as long as the previous one -- like a polite person who knocks on a door, waits, knocks again, waits longer, instead of pounding continuously.

2. Set a maximum retry count. After 5 retries (or whatever is appropriate), mark the job as permanently failed and move it to a dead-letter queue for manual inspection. Jobs that fail forever should not retry forever.

3. Use a separate connection pool for queue workers. The queue worker and the API should not share the same connection pool. Give the queue worker its own smaller pool (say, 3 connections out of a total of 20). Even if the queue worker exhausts its own pool, the API's 17 connections remain available. This is resource isolation -- the same principle as bulkheads on a ship. If one compartment floods, the walls prevent the water from sinking the whole vessel.

Verify: Trigger a queue job failure deliberately. Watch the retry behavior -- it should back off exponentially. After the max retries, the job should land in the dead-letter queue. Throughout this entire process, make API requests. They should respond normally. The API should never become unresponsive because of queue worker behavior.
