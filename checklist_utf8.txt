AI Output Review Checklist

Use this checklist every time you review code generated by an AI agent.
Go through each section before accepting any AI output.

---

1. Correctness

- [ ] The code does what was asked for (re-read your prompt and compare)
- [ ] Happy path works end-to-end with realistic input
- [ ] Edge cases are handled: empty input, null/undefined, boundary values, maximum lengths
- [ ] Error cases return appropriate errors (not silent defaults or empty results)
- [ ] Return types and response shapes match what callers expect
- [ ] Side effects are intentional (database writes, file mutations, network calls)
- [ ] No off-by-one errors in loops, pagination, or array indexing

2. Security

- [ ] No SQL/NoSQL injection: all queries use parameterized statements or safe query builders
- [ ] Authentication is enforced: new endpoints use the same auth middleware as existing protected routes
- [ ] Authorization is enforced: data queries are scoped to the authenticated principal (no IDOR)
- [ ] Input validation exists at the boundary: types, formats, lengths, and allowed values are checked
- [ ] No secrets hardcoded in source (API keys, tokens, passwords, connection strings)
- [ ] URL handling rejects unsafe schemes (javascript:, data:) and blocks SSRF to private IPs
- [ ] Rate limiting is applied to public and write-heavy endpoints
- [ ] Sensitive data is not logged (PII, tokens, request bodies with credentials)

3. Style and Consistency

- [ ] Naming conventions match the project (camelCase vs snake_case, file naming patterns)
- [ ] Import style matches existing files (relative vs absolute, order, grouping)
- [ ] Error handling follows the project pattern (throw vs return, error types, global handler)
- [ ] Async pattern is consistent (no mixing callbacks, .then(), and await in one module)
- [ ] Code organization follows project structure (service/repository layers, file placement)
- [ ] No commented-out code or dead code paths left behind

4. Dependencies

- [ ] No new dependencies added without justification (check package.json / requirements.txt diff)
- [ ] If a new dependency was added: version is pinned, license is compatible, package is actively maintained
- [ ] No duplicate functionality (new library doing what an existing dependency already does)
- [ ] Import paths resolve correctly (no typos, no missing packages)

5. Tests

- [ ] Tests exist for the new or changed code
- [ ] Tests cover the happy path and at least two edge/error cases
- [ ] Assertions are meaningful (not just "status is 200" but also checking response body)
- [ ] No tautological tests (asserting a mock returns exactly what it was configured to return)
- [ ] Tests are independent (no shared mutable state, no order dependence)
- [ ] Test data is realistic (not "test", "foo", "bar" for fields that have format requirements)
- [ ] Tests actually run and pass (execute them, do not trust the AI's claim that they pass)

6. Architecture

- [ ] New code fits existing patterns (does not introduce a second way to do the same thing)
- [ ] No unnecessary abstraction (no interface with one implementation, no factory for one type)
- [ ] Separation of concerns is maintained (HTTP handling, business logic, and data access are not mixed)
- [ ] No premature optimization (caching, batching, or pooling without a measured performance need)
- [ ] Database changes include proper migrations, indexes, and constraints
- [ ] Configuration is externalized (no hardcoded URLs, ports, or feature flags in source)

7. AI-Specific Red Flags

- [ ] No hallucinated imports (packages or modules that do not exist in the project or registry)
- [ ] No hallucinated methods (verify every method call against the actual library version in use)
- [ ] No placeholder values left behind (YOUR_API_KEY, TODO, FIXME, ..., xxx)
- [ ] No copy-paste artifacts (duplicate function bodies, repeated blocks with minor variations)
- [ ] No orphaned code (functions defined but never called, variables assigned but never read)
- [ ] No fabricated comments (comments describing behavior that the code does not actually implement)
- [ ] No version confusion (verify the code works with the library versions pinned in this project)
- [ ] If AI referenced documentation or APIs, verify those references are real and current

---

Quick Decision Framework

After going through the checklist:

- All clear: accept the code, commit with confidence.
- Style issues only: accept and fix in a follow-up pass, or ask the AI to fix with specific instructions.
- Logic or security issues: reject. Write a specific prompt describing exactly what is wrong and what the correct behavior should be. Do not try to manually patch AI output that has fundamental problems.
- Multiple red flags: reject and restart with a better prompt. If the AI consistently fails on this task, decompose it into smaller pieces.
